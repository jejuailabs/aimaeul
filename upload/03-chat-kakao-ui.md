# 03. 카카오톡 스타일 채팅 UI (회원 전용)

## 목표

50~70대 사용자가 별도 학습 없이 사용할 수 있도록, 회원 화면은 카카오톡과 최대한 동일한 사용 흐름을 따른다.
이 채팅은 단순 메신저가 아니라 마을 홈페이지(05 문서)와 실시간으로 연결되는 Live 데이터 소스다.

## 하단 탭 구조 (5개, 고정 네비게이션)

1. **채팅** (`/app/chat`) — 기본 진입 화면
2. **부녀회 홈** (`/app/home`) — 소속 공동체의 요약 화면 (공지, 다가오는 행사, 최근 사진 미리보기)
3. **사진** (`/app/photos`) — 04 문서 참고
4. **게임** (`/app/games`) — 06 문서 참고
5. **내 정보** (`/app/me`) — 프로필, 다크모드 토글, 로그아웃, 새 마을 참여

탭 아이콘/레이블은 최소한으로 단순하게, 큰 터치 영역(최소 44px) 확보.

## 채팅 리스트 (`/app/chat`)

- 소속된 공동체별 오픈채팅방을 리스트로 표시 (카카오톡 채팅 목록과 동일한 레이아웃)
- 각 행: 공동체 프로필 이미지, 공동체 이름, 최근 메시지 미리보기, 최근 메시지 시각, 안 읽은 메시지 뱃지
- Firestore `communities/{id}/messages`를 `orderBy('createdAt', 'desc').limit(1)`로 구독해 미리보기 실시간 갱신

## 개별 오픈채팅방 (`/app/chat/[communityId]`)

- **상단바**: 공동체 이름, 뒤로가기
- **메시지 영역**: 말풍선 UI. 본인 메시지는 우측 정렬 + 노란색 계열, 타인 메시지는 좌측 정렬 + 회색/흰색,
  프로필 사진과 이름 표시(카카오톡과 동일)
- **메시지 타입별 렌더링**:
  - `text`: 일반 말풍선
  - `photo`: 썸네일 + 탭 시 확대, EXIF 오버레이 정보 소량 표시(04 문서 참고)
  - `emoji`: 큰 사이즈로 이모티콘만 표시 (텍스트 말풍선 없이)
  - `game_result`: 카드형 말풍선, 예) "🎲 감자캐기 순번 — ① 김영희 ② 이순자 ③ 박미숙" (06 문서 참고)
  - `system`: 중앙 정렬 회색 텍스트 (예: "OO님이 참여했습니다")
- **실시간 구독**: `onSnapshot`으로 `messages` 컬렉션을 `orderBy('createdAt').limit(최근 N개)`로 구독, 새 메시지 도착 시
  자동 스크롤
- **입력 영역(하단 고정)**:
  - 텍스트 입력창
  - 사진 첨부 버튼 (탭 시 04 문서의 업로드 플로우로 연결)
  - 이모티콘 버튼 (탭 시 이모티콘 패널 오픈 — 아래 참고)
  - 게임 버튼 (탭 시 06 문서의 게임 실행 패널 오픈)
  - 전송 버튼

## 이모티콘 시스템

- `communities/{id}.emojiPackIds` 배열로 등록된 이모티콘 팩을 불러온다
- 이모티콘 팩 데이터 구조: `emojiPacks/{packId}` → `{ name, images: string[] (Storage URL), createdBy, communityId }`
- 기본 제공 이모티콘 세트(서비스 공통, 모든 공동체가 사용 가능) + 운영자가 업로드한 커스텀 팩(9/15/24개 세트)을
  함께 노출
- 운영자 업로드 플로우: PNG(투명 배경) 여러 장을 한 번에 업로드 → Storage에 저장 → `emojiPacks` 문서 자동 생성
- 회원은 버튼 탭만으로 선택 → `messages` 컬렉션에 `type: "emoji"`, `text: <이미지URL>`로 전송

## 구현 시 주의사항

- "새 채팅 시작" 같은 개념은 존재하지 않는다. 사용자는 소속 공동체의 고정 오픈채팅방에서만 대화한다 (1:1 DM은
  이 명세서 범위 밖 — 별도 지시 없이 구현하지 않음)
- 메시지 전송 시 낙관적 업데이트(optimistic UI)로 즉시 화면에 반영 후 Firestore 쓰기, 실패 시 재전송 UI 제공
- 채팅 메시지는 곧바로 05 문서의 마을 홈페이지 Live Chat에도 동일하게 노출되므로, 부적절한 콘텐츠 필터링(욕설 등)은
  최소한의 클라이언트 사이드 필터라도 고려 (고도화는 범위 밖, TODO로 표시만 해둘 것)
