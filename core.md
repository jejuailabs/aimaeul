## 1. 프로젝트 한 줄 정의

카카오톡 UI/UX를 그대로 차용한 **마을 공동체(부녀회·청년회·노인회·동호회 등) 디지털 플랫폼**.
사용자는 새로운 앱 사용법을 배울 필요 없이 카카오톡처럼 사용하지만, 그 활동(채팅·사진·게임 결과)이
실시간으로 마을 홈페이지에 반영되고, AI가 이를 자동으로 정리해 "AI가 읽을 수 있는 디지털 마을 아카이브"로
축적되는 것이 핵심 차별점입니다.

## 2. 타깃 사용자

- **주 타깃**: 50~70대, 카카오톡 외 다른 앱 UX에 익숙하지 않은 마을 공동체 구성원
- **부 타깃(비회원)**: 대한민국 지도에서 마을을 둘러보는 일반 방문자, 관광객, AI/검색엔진 크롤러

## 3. 기술 스택 (확정)

| 영역 | 선택 |
|---|---|
| 프론트엔드 | Next.js (App Router), 배포는 Vercel |
| 백엔드/DB | Firebase (Firestore, Firebase Storage, Firebase Auth, Cloud Functions) |
| 인증 | Google 로그인 (Firebase Auth) 단일 방식 |
| 다크/라이트 모드 | 시스템 설정 자동 감지 + 사용자가 수동 토글 가능한 스위치, 선택값은 로컬 저장 + 로그인 시 Firestore 사용자 문서에도 저장 |
| 실시간 동기화 | Firestore `onSnapshot` 실시간 리스너 (채팅 ↔ 홈페이지 Live Chat 동기화의 핵심) |
| 이미지 처리 | 업로드 시 Cloud Functions(or Vercel Edge Function)에서 EXIF 파싱 |
| AI 기능 | Anthropic API(Claude) 연동 — 일일신문/월간기록/AI검색/스토리 생성 등 |

세부 아키텍처와 Firestore 스키마는 `docs/01-architecture.md` 참고.

## 5. 구현 우선순위 (Phase)

- **Phase 1 (MVP 뼈대)**: 01 → 02 → 03. 로그인, 마을 선택, 카카오톡 스타일 채팅 UI까지가 가장 먼저 동작해야 함.
- **Phase 2 (핵심 차별점)**: 04 → 05. 사진 EXIF 자동화 + Live Feed + 마을 홈페이지 실시간 반영. 이 프로젝트의 정체성.
- **Phase 3 (커뮤니티 도구)**: 06 → 07. 게임/추첨 도구, 제보하기, 빈집소개.
- **Phase 4 (AI 플랫폼)**: 08. AI 자동 편집/신문/검색/공개 API. Phase 1~3의 데이터가 쌓인 뒤 의미가 커짐.

에이전트는 반드시 이 순서를 지킬 필요는 없지만, Phase 2가 Phase 1의 데이터 구조(채팅 메시지, 사용자, 마을 컬렉션)에
의존하므로 최소한 01~03은 먼저 구현되어야 합니다.

## 6. 공통 원칙 (모든 문서에 적용)

- **UI 원칙**: 새로운 인터페이스를 발명하지 않는다. 항상 "카카오톡이라면 어떻게 생겼을까"를 기준으로 삼는다.
  큰 버튼, 최소 입력, 하단 탭 5개, 말풍선 중심 인터랙션.
- **입력 최소화**: 사용자가 텍스트를 직접 입력하는 지점을 최소화한다. 사진 업로드, 버튼 탭이 우선.
- **실시간성**: 채팅 ↔ 홈페이지는 항상 실시간(Firestore listener)으로 연결되어야 하며, 새로고침 없이 반영되어야 한다.
- **다크/라이트 모드**: 모든 화면(회원 채팅형 UI, 비회원 홈페이지 UI 포함)에 공통 적용. 토글 스위치는 "내 정보" 탭과
  비회원 홈페이지 상단 두 곳에 존재.
- **확장성**: 모든 데이터 모델은 부녀회 외에 청년회/노인회/동호회 등 임의의 "공동체(community)" 단위로 확장 가능하게
  설계한다 (하드코딩된 "부녀회" 문자열 금지, `communityType` 필드로 관리).

## 7. 에이전트 작업 방식 가이드

- 각 `docs/0X-*.md`는 "목표 → 데이터 모델 → 화면/라우트 → 컴포넌트 → 구현 시 주의사항" 구조로 되어 있습니다.
- 문서 간 데이터 모델이 중복 언급되는 경우, `docs/01-architecture.md`의 정의가 최종 기준(source of truth)입니다.
- 명세서에 없는 세부 디자인(정확한 색상 코드, 폰트 등)은 에이전트가 프로젝트의 톤(카카오톡 느낌: 노란색 포인트 컬러,
  둥근 말풍선)에 맞게 합리적으로 결정해도 됩니다.
- 명세서에 없는 기능을 임의로 추가하지 마세요. 특히 결제, 알림톡 연동 등은 이 문서 세트에 없으므로 별도 지시 전까지 구현하지 않습니다.
